Topic Clustering reads the keyword clusters already in your library and sorts them into a small set of broader subject areas called **themes**. The result is a tidy map you can plan content around: a handful of themes at the top, and the keyword clusters that belong under each one.

## The three levels in RankEarly

Before anything else, it helps to be clear about the words we use. RankEarly organizes everything into three levels:

### A note on the word "topic"

The word *topic* means different things across SEO tools — Semrush, Ahrefs, and Keyword Insights each use it a little differently, sometimes for a single keyword, sometimes for a cluster, sometimes for a subject area. In RankEarly, **topic is just another word for keyword cluster** — a page-level opportunity. This page uses *keyword cluster* throughout to avoid confusion.

## When to use it

Topic Clustering becomes useful once your library has grown too large to scan cluster by cluster. Run it when:

- your library has enough keyword clusters that a top-down map would save time
- you want to decide which subject areas are worth investing in and which to skip
- you want a cleaner site architecture by grouping pages under a few categories

You do **not** need themes to start. For a new site or a small library, pick keyword clusters directly from your [library](/keyword-library) and start writing. Themes become worth the step once you have enough clusters that grouping them pays off — typically a few dozen.

A typical 500–1,000-keyword library produces around 15–20 themes. Themes are intentionally selective so each one is substantial enough to build content around.

## Before you start

- Choose a keyword library that belongs to the same project
- Make sure the library already contains keywords with SERP snapshots — the existing keyword clusters are the input
- Use [Keyword Expansion](/research/keyword-expansion) first if the project does not have a usable library yet

## Reading the results

Review each theme as a group of related keyword clusters. Review each keyword cluster as one page opportunity — a group of closely related searches Google answers with the same results.

Theme names come from the most central keyword cluster inside that theme. Treat them as working labels, not final page titles.

Use the cluster metrics and the underlying keywords to decide:

- which themes best align with your project and expertise
- which keyword clusters are the best places to start writing
- which opportunities should move into a [content brief](/planning/content-brief)

## One page, one keyword cluster

A keyword cluster is the minimum unit for content planning. Writing multiple pages for the same keyword cluster creates cannibalization — Google already treats those queries as the same question, so two of your pages end up competing with each other.

Plan one page per keyword cluster. If a cluster feels too narrow to fill a page, it usually belongs inside a broader cluster; re-check the clusters in your library before splitting.

## Why themes matter

Themes do two concrete things keyword clusters alone do not:

1. **A bird's-eye view of your content plan.** Instead of staring at dozens of clusters in a flat list, you see a handful of subject areas — each one a section you can sequence, prioritize, or use to build breadcrumb-style navigation. You also see at a glance where your library is strong and where it is thin.
2. **A starting point for building linked groups of pages.** Inside each theme, one keyword cluster usually has broader scope than the others. That cluster is a natural candidate for an overview page, with the remaining clusters in the theme as detailed pages linking up to it. This pattern is called the [pillar-cluster model](/glossary#pillar-cluster-model) — themes give you the map to build it.

## Themes are based on meaning, not word overlap

RankEarly builds themes around what each keyword cluster actually means for your business. Most other tools' "topic cluster" or "parent topic" features rely on linguistic similarity — NLP models that group phrases sharing close wording, stems, or related terms. That approach is well-established and fast, but at the theme level it can group queries that *sound* related yet answer different questions for the reader. For example, `what is CRM` and `what is CRM account` are linguistically close but are distinct page-level topics.

Because RankEarly groups by business meaning instead, themes come out more selective: a 500–1,000-keyword library typically yields ~15–20 themes, and clusters that do not clearly fit any theme stay uncategorized rather than being pushed into a near-match.

## Where themes are saved

Themes are saved to the keyword library. Re-running Topic Clustering on the same library replaces the previous themes; keyword clusters are not affected.

## What to do with the results

- Keep the themes that match your product, audience, and expertise
- Start with keyword clusters that combine lower KD with stronger search volume
- Create a [content brief](/planning/content-brief) from the keyword clusters you plan to write next
- Use the themes to plan an overview page (a [pillar page](/glossary#pillar-page)) once you have enough supporting cluster content

## Tips

<AccordionGroup>
  <Accordion title="Topic Clustering depends on your library quality">
    Themes are only as good as the clusters they group. If the library is thin or outdated, the map of themes will be thin too.
  </Accordion>
  <Accordion title="Build the library from multiple angles before clustering">
    Themes are strongest when the library already covers different angles of the same market. Run Keyword Expansion across features, audiences, and business angles first, then run Topic Clustering on that fuller library.
  </Accordion>
  <Accordion title="Re-running replaces the previous themes">
    Topic Clustering does not keep multiple theme sets for the same library. If you want a different map, duplicate or create a separate library first.
  </Accordion>
